## 1. Thin Provisioning (씬)

### 정의

- 논리적 디스크 용량만 할당
    
- 실제 물리 스토리지는 **데이터 기록 시점에만 할당**
    

### 특징

- Datastore 사용량이 실제 데이터 크기에 비례
    
- 오버프로비저닝 가능
    
- 스토리지 활용률이 높음
    

### 장점

- 초기 스토리지 소모 최소화
    
- VM 대량 생성에 유리
    
- 개발/테스트 환경 적합
    

### 단점 (핵심 리스크)

- Datastore 고갈 시 VM 쓰기 실패
    
- 운영 중 장애 가능성 높음
    
- 성능 예측 어려움
    
- 모니터링 필수
    

### 권장 사용

- Dev/Test
    
- 단기 VM
    
- 사용량 예측 가능한 워크로드
    

---

## 2. Thick Provisioning (씩)

### 정의

- 디스크 생성 시 **전체 용량을 즉시 물리 스토리지에 할당**
    

### 유형

- **Lazy Zeroed**: 공간만 즉시 할당, 초기화는 최초 쓰기 시
    
- **Eager Zeroed**: 생성 시 모든 블록 초기화 (FT 필수)
    

### 특징

- Datastore 공간 즉시 감소
    
- 오버프로비저닝 불가
    
- 성능 및 안정성 예측 가능
    

### 장점

- 스토리지 고갈 리스크 없음
    
- 안정적인 I/O 성능
    
- 장애 원인 단순화
    

### 단점

- 스토리지 낭비 가능
    
- 초기 디스크 생성 시간 증가
    

### 권장 사용

- 운영(Production)
    
- DB, 로그, 메시지 서버
    
- 장애 허용 범위가 낮은 시스템
    

---

## 3. 핵심 차이 비교

|구분|Thin|Thick|
|---|---|---|
|공간 할당|사용 시점|생성 시점|
|스토리지 효율|높음|낮음|
|성능 안정성|낮음|높음|
|운영 리스크|높음|낮음|
|권장 환경|Dev/Test|Production|

---

## 4 운영 판단 한 줄 기준

- **장애 나면 안 되면 → Thick**
    
- **효율이 더 중요하면 → Thin**
    
- **운영 책임자가 불안하면 → Thick**
    

---

## 5. 요약

- Thin은 **효율 중심**, Thick은 **안정 중심**
    
- 기술 선택 문제가 아니라 **운영 책임 문제**
    
- ESXi 운영 환경에서는 기본값을 Thick으로 두는 것이 보수적임.
