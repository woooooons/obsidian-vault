# Semantic Versioning (SemVer) 규약

**Semantic Versioning(유의적 버전)** 은 소프트웨어의 버전 번호를 어떻게 부여하고 관리할지에 대한 세계적인 표준 규약. 버전 번호만 보고도 이 업데이트가 기존 코드에 어떤 영향을 주는지 개발자가 직관적으로 파악할 수 있게 하는 것이 목적.

## 2. 버전 구조: [MAJOR].[MINOR].[PATCH]

버전 번호는 반드시 세 가지 숫자의 조합으로 구성. (예: `1.2.3`)

### ① MAJOR (주 버전)

- 기존 버전과 **호환되지 않는** API 변경이 발생했을 때 올림.
    
- 이 숫자가 올라가면 해당 라이브러리를 사용하는 사용자는 코드를 수정해야 할 가능성이 높음 (Breaking Changes).
    
- **MAJOR가 올라가면 MINOR와 PATCH는 `0`으로 초기화. (예: `1.5.2` -> `2.0.0`)
    

### ② MINOR (부 버전)

- 기존 버전과 **호환되면서** 새로운 기능을 추가했을 때 올림.
    
- 기존에 쓰던 기능들은 그대로 동작하므로, 사용자는 안심하고 업데이트할 수 있음.
    
- MINOR가 올라가면 PATCH는 `0`으로 초기화. (예: `1.5.2` -> `1.6.0`)
    

### ③ PATCH (수정 버전)

- 기존 버전과 **호환되면서** 단순한 버그 수정(Bug fixes)만 이루어졌을 때 올림.
    
- 기능 변화는 없으며 안정성만 향상된 상태.
    

---

## 3. 핵심 원칙 (Rules)

1. **0.y.z (초기 개발):** 버전이 `0`으로 시작하는 경우(예: `0.1.0`)는 아직 초기 개발 단계임을 의미. 이때는 아무 때나 API를 변경해도 무방하며, 아직 안정화되지 않았음을 나타냄.
    
2. **1.0.0 (정식 릴리즈):** 소프트웨어가 실제 운영 환경에서 쓰일 준비가 되었다면 `1.0.0`으로 시작. 이때부터 위 규약이 엄격하게 적용.
    
3. 한번 배포된 버전의 내용은 절대 수정하지 않음. 수정이 필요하면 무조건 새 버전을 발행해야 함.
    
4. MAJOR > MINOR > PATCH 순으로 중요도를 가짐.
    

---

## 4. 추가 식별자 (선택 사항)

정식 버전 뒤에 하이픈(`-`)을 붙여 특수한 상태를 표시할 수 있음.

- **Pre-release:** 정식 배포 전 테스트 버전을 의미.
    
    - 예: `1.0.0-alpha`, `1.0.0-beta`, `1.0.0-rc.1` (Release Candidate)
        
- **Build Metadata:** 빌드 관련 정보를 플러스(`+`) 기호 뒤에 추가할 수 있음. (우선순위에는 영향 없음)
    
    - 예: `1.0.0+20240121`
        

---

## 5. 요약

> **"기존 기능을 망가뜨리면 MAJOR, 기능을 추가하면 MINOR, 버그만 잡으면 PATCH를 올린다."**
